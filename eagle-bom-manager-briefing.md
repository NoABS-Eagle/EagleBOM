# Eagle BOM Manager — Бриф проекта

## Контекст и мотивация

Autodesk похоронил Eagle, но инструмент живёт у ветеранов. Цель — показать что Eagle можно
допилить и он будет лучше многих современных ECAD. BOM-менеджер — первая и самая болезненная
точка. Аудитория: оставшееся сообщество Eagle-пользователей. Проект называется **NoABS**
(No Autodesk BullShit).

---

## Проблема

В Eagle нет нормального BOM-менеджера с групповым редактированием атрибутов.

Текущий процесс пользователя:
1. Экспортировать BOM из Eagle
2. Вручную смотреть на атрибуты и находить несоответствия
3. Запустить самописный ULP-скрипт "скопировать атрибуты" — задать атрибут одному компоненту,
   скрипт копирует его всем с тем же value+footprint

Проблема скрипта: нет визуального слоя — не видно групп, не видно где заполнено/не заполнено,
процесс непрозрачный.

ULP не позволяет сделать нормальный spreadsheet UI — это потолок платформы.

---

## Решение

**Внешний веб-инструмент**, работающий напрямую с Eagle .sch файлом (XML).

### Почему веб:
- Eagle .sch — это XML, парсится через DOMParser прямо в браузере
- Деплой на GitHub Pages — работает у всех без установки
- Платформонезависимо (Windows/Mac/Linux)
- Производительность не критична (просто таблица, не 3D)
- Никакого Python/pyinstaller/зависимостей у пользователя
- Полностью serverless — парсинг, редактирование, генерация файлов — всё в браузере

### Двойная роль инструмента:
Инструмент — это одновременно **редактор BOM** и **вьювер BOM**.
Отдельный BOM-файл (Google Sheets, Excel) перестаёт быть нужным вообще.
Разработчик просто даёт ссылку на менеджер с открытым .sch — закупщик видит актуальный BOM.
Экспорт .xlsx доступен если нужно (SheetJS, генерируется на клиенте).

### Workflow:
```
Drag & drop .sch → BOM-редактор в браузере → редактирование атрибутов →
скачать .scr → запустить .scr в Eagle → Eagle применяет всё сам
```

---

## Архитектура: .sch только для чтения

**Инструмент читает .sch но никогда не пишет в него.**

Eagle синхронизирует атрибуты между .sch и .brd через живой канал между открытыми окнами.
Прямая запись в .sch файл снаружи нарушила бы консистентность схемы и платы.

Вместо этого — генерация **.scr скрипта** (Eagle Script): простой текстовый файл с командами
CLI для Eagle. Пользователь запускает его в открытой схеме, Eagle применяет атрибуты сам —
и в схему, и в плату автоматически через живой канал.

Формат .scr:
```
ATTRIBUTE C1 MPN 'GRM155R71C104KA88D';
ATTRIBUTE C5 MPN 'GRM155R71C104KA88D';
ATTRIBUTE C12 MPN 'GRM155R71C104KA88D';
```

Генерируется тривиально из данных BOM-менеджера. Единственный выходной артефакт.

## Логика группировки

Компоненты объединяются в одну строку BOM по **совпадению ВСЕХ атрибутов** (не только value+footprint).
Именно так работает штатный экспорт BOM из Eagle — берём ту же логику.

Колонки таблицы **динамические** — парсятся из атрибутов компонентов в схеме.
Сколько уникальных атрибутов нашлось в .sch — столько колонок. Лишние колонки (где у компонента
нет такого атрибута) просто остаются пустыми.

---

## Логика подсветки (предупреждения)

**Жёлтая подсветка строки** — если компоненты с одинаковыми value+footprint попали в разные
группы из-за расхождения в других атрибутах.

Правило одно, stateless, без исключений:
```
Если value+footprint одинаковые, но компоненты в разных группах → жёлтый
```

Частные случаи которые покрываются автоматически:
- Все атрибуты одинаковые → не жёлтый
- MPN не задан ни у кого → не жёлтый (просто ещё не заполнено)
- MPN задан частично (у одних есть, у других нет) → жёлтый
- Два компонента намеренно с разными MPN → жёлтый, пользователь видит и игнорирует

**Никаких всплывающих окон и блокирующих диалогов.** Только цвет. Пользователь оценивает
сам и игнорирует если всё правильно. Хранить "подтверждение намеренности" нигде не нужно —
логика полностью stateless.

---

## Редактирование

Редактирование любого атрибута на уровне **группы**:
- Пользователь меняет значение в строке группы
- Значение применяется ко **всем компонентам группы**
- До скачивания .scr пользователь может свободно редактировать и передумывать

---

## Применение изменений

**Явный commit** — кнопка **"Скачать .scr"**.

Скачивается единственный файл — .scr скрипт со всеми изменёнными атрибутами.
Пользователь запускает его в Eagle → Eagle сам всё применяет в схему и плату.

---

## Атрибуты компонентов (рекомендуемый набор)

Инструмент не навязывает набор атрибутов — он динамический. Но типичный рабочий набор:
- **VALUE** — номинал (100nF, 10k и т.д.)
- **PACKAGE** — корпус (0402, SOT-23 и т.д.)
- **MANUFACTURER** — название производителя
- **MPN** — артикул производителя
- **LCSC** — артикул в магазине LCSC

Это не дублирование: MPN идентифицирует компонент глобально, LCSC — операционный артикул
для конкретного процесса заказа.

---

## Что НЕ входит в v1

- **DNP / варианты сборки** — касается и .brd, отдельная тема
- **Интеграция с магазинами** — привязывать инструмент к магазину плохая идея
- **Кликабельные ссылки на MPN** — пользователь уже сам открыл страницу компонента заранее

---

## Минимальный набор функций v1

1. **Drag & drop** .sch файла
2. **Таблица** с динамическими колонками из атрибутов схемы
3. **Группировка** по совпадению всех атрибутов
4. **Жёлтая подсветка** подозрительных групп (same value+footprint, разные атрибуты)
5. **Inline редактирование** атрибута на уровне группы → применяется ко всем в группе
6. **Экспорт .xlsx** — SheetJS, генерируется на клиенте, для закупщиков
7. **Кнопка "Скачать .scr"** — Eagle скрипт который применяет все изменения атрибутов

## Технический стек

- Чистый HTML + JS (или лёгкий фреймворк)
- DOMParser для парсинга Eagle XML
- SheetJS для экспорта .xlsx
- Деплой: GitHub Pages
- Полностью serverless — никаких серверных зависимостей
